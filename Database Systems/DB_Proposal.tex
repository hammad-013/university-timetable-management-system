\section{Introduction}

In any education system, time management is very important for both teachers and
students. A managed schedule helps everyone stay organized and focused. Therefore, managing university timetables manually through spreadsheets and emails is a time consuming and error-prone process. Moreover, changing a class becomes a problem with manual process. The University Timetable Management System is a web-based application that solves these problems by centralizing timetable creation and management for university. It allows administrators to create and publish timetables, faculty to view their assigned schedules and submit schedule change requests, and students to access their class timetables all from a single platform.

\section{Problem Statement}

The current manual timetable process suffers from the following issues:

\begin{itemize}
    \item Venue conflicts: The same classroom is accidentally assigned to two courses at the same time.
    \item Instructor conflicts: A faculty member is scheduled for two courses simultaneously.
    \item No central system: Timetable data is scattered with no single source.
    \item No Change Request Mechanism: Faculty have no formal way to request changes to their assigned schedule, forcing them to make use of emails or verbal communication.
\end{itemize}

These problems lead to schedule mishaps, wastage of administrative time. To overcome these gaps, a database based system with role based access and constraints should be implemented.

\section{Objectives}
The key objectives of the University Timetable Management System are \cite{srs}:

\begin{enumerate}
    \item Design a normalized relational database schema for all entities.
    \item Implement venue and instructor conflict detection.
    \item Enforce Role-Based Access Control for Admin, Faculty, and Student users.
    \item Enable administrators to create and publish timetables.
    \item Allow faculty and students to view their respective schedules.
    \item Enable faculty to submit schedule change requests for administrator review.
\end{enumerate}

\section{Scope}

The following features are included in the system \cite{srs}:

\begin{itemize}
    \item \textbf{Timetable Management:} Create, edit, delete, and publish timetables and export to CSV.
    \item \textbf{Conflict Detection:} Venue double-booking and instructor overlap detection.
    \item \textbf{User Management:} Registration, login, RBAC, profile management, user deactivation.
    \item \textbf{Resource Management:} Venue and lab database with conflict prevention.
    \item \textbf{Course Management:} Course registry linked to departments and instructors.
    \item \textbf{Dashboards:} Role-specific views for Admin, Faculty, and Student.
\end{itemize}

The following are explicitly excluded:

\begin{itemize}
    \item Automatic timetable generation.
    \item Timetable archiving and versioning.
    \item All reporting (summary, utilization, audit log).
    \item Soft-delete, MFA, bulk user import, timetable import, scheduled/in-app notifications, conflict resolution suggestions.
\end{itemize}

\section{Data Collection}

Data required to test the system will be collected as follows:

\begin{itemize}
    \item User data (faculty and students): obtained from kaggle/mendeley data and github.
    \item Course and department information: collected from university's website.
    \item Venue and lab details: manually created.
    \item Academic calendar and time slot configuration: set up by the Admin based on the university schedule.
    \item Sample/test data: synthetic data generated using Python(Faker).
\end{itemize}

\section{System Functionality}
The following functionalities describe the core operations the system will support, based on the defined scope \cite{srs}.

\begin{enumerate}
    \item Administrators can create, edit, delete, and publish timetables by assigning courses, instructors, venues, and time slots.
    \item The system detects and does not let the administrator select a venue or instructor at the same time slot.
    \item Administrators can manage user accounts including creation, deactivation, and role assignment.
    \item Users log in with email and password and their access is restricted based on their role.
    \item Faculty can view only their assigned courses and schedule.
    \item Faculty can submit a schedule change request specifying the request type, preferred time slot, venue, and reason.
    \item Students can view only their enrolled class timetable.
    \item Administrators can manage venues, labs and courses.
    \item Administrators can review pending schedule change requests and approve or reject them.
    \item Timetable data can be exported to CSV.
\end{enumerate}

\section{Conclusion}

The University Timetable Management System will replace the existing manual and unstable scheduling process with an organized database-based system. The University Timetable Management System will decrease errors in scheduling and administration overhead due to enforcing role-based access, conflict detection, and centralized data management. The project scope is suitable as it relates to the course Database Systems because it offers practical implementation of concepts such as schema normalization, relational integrity, role-based access control and constraint enforcement.
